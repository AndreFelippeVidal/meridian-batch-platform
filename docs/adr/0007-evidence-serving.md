# ADR 0007 — Evidence for BI serving, deployed to Vercel

## Status
Accepted

## Context
The platform needs a visible front-end artifact that non-technical reviewers (recruiters,
hiring managers) can open in a browser. Options considered: Metabase, Superset, Streamlit,
Evidence.

## Decision
Use Evidence (`@evidence-dev/evidence`) for the dashboard layer.

Rationale:
- **SQL-native:** analysts write SQL directly in Markdown — no drag-and-drop BI config
  to maintain.
- **Static output:** `npm run build` produces a plain HTML/JS bundle deployable anywhere;
  no server required.
- **DuckDB connector built-in:** Evidence reads Parquet files generated by
  `npm run sources` (which runs SQL against the DuckDB file) — the dashboard depends on
  pre-built source files, not a live DuckDB connection at serve time.
- **Vercel deploy:** zero-config static site hosting; the `serving/` directory is the
  Vercel project root.

Four pages: Overview (KPIs + GMV trend), Products (category revenue, margin), Customers
(new-customer trend, segment analysis), Data Quality (GMV integrity, payment failure rate).

**Workflow:** `make evidence-sources` must run after `make transform` to regenerate
source Parquet files before `make evidence-build`. The build is gitignored; the source
`.sql` query files and `connection.yaml` are committed.

## Consequences
- Dashboard data is a snapshot — not live. Acceptable for a portfolio demo where freshness
  is controlled by the operator running `make evidence-sources`.
- The `needful_things` demo source is scaffolded but unused; its DuckDB file is committed
  as part of the Evidence template (small, ~200 KB).
- Vercel redeploy is a one-command operation once authenticated: `npx vercel --prod`
  from `serving/`.
